Chapter 01

核心概念一览

LangGraph 的设计哲学很简单——把 AI 工作流建模为有向图。不同于线性链式调用，图结构天然支持循环、条件分支和持久化状态，这正是构建真正智能 Agent 的关键。

◈

State 状态

贯穿整个图的共享数据结构，每个节点都能读取和更新它。就像 Agent 的"工作记忆"。

⬡

Node 节点

执行具体工作的函数——调用 LLM、执行工具、运行业务逻辑。接收 State，返回更新。

⟶

Edge 边

定义节点之间的连接和流向。普通边直接连接，条件边根据 State 动态选择路径。

Chapter 02

State：图的共享记忆

State 是一个 TypedDict，它在所有节点之间流动，携带 Agent 需要的所有信息。每个节点只需返回它想更新的字段——其余保持不变。

📌 基础定义

Python from typing import TypedDict, Annotated import operator class State(TypedDict): # 无 Reducer → 每次直接覆盖 user_query: str # 有 Reducer → 新值追加到列表末尾 messages: Annotated[list, operator.add] # 自定义 Reducer → 完全控制合并逻辑 context: Annotated[dict, merge_dicts]

🔄 Reducer 合并策略

Reducer 决定了当多个节点更新同一个字段时，值如何合并。这在并行执行场景中尤为重要。

Reducer 示例 # 覆盖（默认）：新值替换旧值 user_query: str # State: "Hello" → Node 返回 "World" → State: "World" # 追加：使用 operator.add messages: Annotated[list, operator.add] # State: ["Hi"] → Node 返回 ["Hello"] → State: ["Hi", "Hello"] # 自定义：任意合并函数 def reduce_list(left, right): """去重合并""" return list(set(left + right))
Chapter 03

Node：执行具体操作

节点就是普通的 Python 函数。它接收当前 State，做一些操作，然后返回一个字典表示要更新的字段。LangGraph 不限制你在函数里做什么。

LLM Node def chatbot(state: State): """调用大语言模型进行推理""" response = llm.invoke(state["messages"]) return {"messages": [response]} # LLM 节点是 Agent 的"大脑" # 它分析对话历史，决定下一步行动 # 可能直接回答，也可能请求调用工具
Tool Node from langgraph.prebuilt import ToolNode tools = [search_tool, calculator_tool] tool_node = ToolNode(tools) # 或者手动实现： def tool_executor(state: State): """执行 LLM 请求的工具调用""" last_msg = state["messages"][-1] results = [] for call in last_msg.tool_calls: result = execute(call) results.append(result) return {"messages": results}
Custom Node def validate_input(state: State): """自定义业务验证逻辑""" query = state["user_query"] if len(query) < 3: return {"error": "输入太短"} if contains_sensitive(query): return {"error": "包含敏感信息"} return {"validated": True} # 自定义节点可以实现任意逻辑 # 数据库查询、API 调用、文件处理...
Chapter 04

Edge：控制流程走向

边定义了"做完这步之后，下一步去哪里"。LangGraph 提供三种边，其中条件边是最强大的——它赋予 Agent 动态决策能力。

➡️ 普通边 (Normal Edge)

A 完成后，无条件走到 B。

builder.add_edge("node_a", "node_b") builder.add_edge(START, "chatbot") # 入口 builder.add_edge("tools", END) # 出口

⛏️ 条件边 (Conditional Edge)

根据当前 State 动态决定下一个节点。这是 Agent 智能行为的关键。

def route(state: State) -> Literal["tools", "__end__"]: last_msg = state["messages"][-1] if last_msg.tool_calls: return "tools" # LLM 想调工具 → 去工具节点 return END # 不需要工具 → 结束 builder.add_conditional_edges("chatbot", route)

🔀 并行分支

一个节点有多条出边时，目标节点并行执行。

# node_a 完成后，node_b 和 node_c 同时执行 builder.add_edge("node_a", "node_b") builder.add_edge("node_a", "node_c") # 两个节点的输出通过 Reducer 合并到 State
Chapter 05

执行流程：ReAct Agent

下面是最经典的 Agent 模式——ReAct（推理 + 行动）循环。LLM 先推理，决定要不要调用工具；调用后把结果送回 LLM，继续推理，直到不需要工具为止。

▶ START
◈ LLM 节点（推理）
⟐ 条件判断：需要工具？
✓ 是
⬡ 工具节点（执行）
↻ 结果回传 LLM
✗ 否
■ END

⚡ Superstep 执行模型

LangGraph 受 Google Pregel 启发，以"超级步骤"推进执行。在每个 Superstep 中，所有收到消息的活跃节点并行运行。当所有节点都空闲且没有消息在传递时，图终止。这个模型既高效又可预测。

Chapter 07

高级特性

这些能力让 LangGraph 从原型走向生产。

Checkpointing 检查点
每一步自动保存状态快照。支持故障恢复（从断点继续）、时间旅行（回到任意历史步骤）、调试（审视每一步的完整状态）。使用 MemorySaver（内存）或自定义持久化后端。
Human-in-the-Loop 人类介入
在关键节点暂停执行，等待人类审核。用 interrupt_before/interrupt_after 指定暂停点。操作员可以查看当前状态、修改数据，然后恢复执行。非常适合合规审查和质量控制。
Streaming 流式输出
原生 token 级别流式传输，实时展示 Agent 的推理过程。用户不再面对空白等待，而是看到 Agent "思考" 的每一步。支持流式返回中间状态和事件。
Subgraph 子图
在一个节点中嵌入另一个完整的图。实现模块化——把复杂逻辑封装为子图，在多个父图中复用。子图有自己的 State，通过输入/输出映射与父图通信。
Middleware 中间件 (v1.0+)
在模型调用前后插入钩子函数。before_model 可以修改输入、做审核；after_model 可以后处理输出、记录日志。在不改动节点代码的情况下横切关注点。
Chapter 08

应用领域

LangGraph 几乎适用于所有需要 多步推理、工具调用和状态管理 的 AI 应用场景。以下是最典型的领域：

🤖

智能客服系统

Customer Support Agent

多轮对话理解用户意图，自动查询知识库、调用工单系统，在必要时转接人工。支持上下文记忆，跨对话保持用户偏好。

条件路由 人机协作 多工具调用
📊

数据分析 Agent

Data Analysis Pipeline

接收自然语言查询，自动生成 SQL / Python 代码，执行查询，分析结果并生成可视化报告。迭代优化直到用户满意。

代码生成 循环优化 工具链
📝

内容创作工作流

Content Creation Workflow

多 Agent 协作：研究 Agent 搜集资料 → 写作 Agent 生成草稿 → 编辑 Agent 润色修改 → 审核 Agent 质量把关。层级式架构管理。

多 Agent 子图 并行执行
💼

企业流程自动化

Enterprise Automation

审批流程、文档审核、合规检查。结合 Human-in-the-Loop 在关键节点暂停等人类审批。Checkpointing 保证长流程不丢状态。

持久化执行 人类审批 合规
🔬

RAG 增强检索

Retrieval-Augmented Generation

自适应 RAG：先检索 → 判断质量 → 不够好则重写 query 再检索 → 循环直到找到高质量文档 → 生成回答。迭代式比一次性检索效果好很多。

自适应检索 质量评估 循环
💻

代码开发助手

Coding Agent

理解需求 → 生成代码 → 运行测试 → 分析错误 → 修复 → 重新测试。ReAct 循环天然适合这种"尝试-反馈-改进"的开发模式。

ReAct 循环 自我修复 测试驱动
🏥

医疗辅助决策

Medical Decision Support

症状分析 → 查询医学知识库 → 初步判断 → 人类医生审核确认 → 生成报告。Human-in-the-Loop 保证安全性，检查点保证可追溯。

安全关键 人类审核 可追溯
📧

智能邮件处理

Email Automation Agent

自动分类邮件 → 提取关键信息 → 草拟回复 → 安排日历 → 创建待办。多 Agent 各司其职，Supervisor 统筹调度。

Supervisor 多工具 自动化
🎮

游戏 NPC / 互动叙事

Interactive Storytelling

NPC 有持久记忆（长期 Memory），根据玩家行为动态调整对话和行为。图结构自然表达分支剧情和状态转换。

长期记忆 状态机 个性化

💡 何时选择 LangGraph？

当你需要精确控制 Agent 行为时——循环推理、条件分支、状态持久化、人类介入、生产级部署。如果只是简单的线性 LLM 调用链，LangChain Chains 可能更轻便；如果想快速搭建多 Agent 团队且不需要精细控制，CrewAI 更友好。LangGraph 的定位是生产级 Agent 基础设施。
